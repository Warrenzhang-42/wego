# WeGO - AI 本地导览产品方案文档

> **愿景**：让每一次旅行都成为一场跨越时空的文化深度对话。

## 1. 产品概述
**WeGO** 是一款定位于"下一代 AI 智能游伴"的移动端旅行应用。它打破了传统旅游 App 仅提供"信息堆砌"的局限，通过大语言模型（LLM）与地理位置服务（LBS）的深度融合，为用户提供私人口语化、具备深厚文化底蕴的本地导览服务。

### 1.1 产品双锚定位
WeGO 的产品价值锚定在两个核心维度的交汇点上，这也是区别于所有竞品的根本差异：

| 维度 | 定义 | 竞品现状 | WeGO 实现 |
| :--- | :--- | :--- | :--- |
| **情绪价值（AI 陪伴）** | 让旅行不再孤单、不再无聊 | 工具化的导航/攻略，无情感连接 | 人格化 AI 导游，口语化讲解，知你所好 |
| **实用价值（LBS 导览）** | 真实解决"去哪、怎么走、看什么" | 信息堆砌，缺乏即时性与深度 | 空间感知讲解，实时路线规划，深度知识库 |

> 当"情绪价值"与"实用价值"在同一个交互界面中无缝融合时，用户获得的不是一个工具，而是一个**有温度的文化旅行搭档**。

### 1.2 产品核心价值 (Core Value)
*   **去攻略化**：无需由于信息过载而产生的复杂预调研，AI 个性化定制路线，随走随看。
*   **深层次体验**：挖掘城市背后的非遗、历史与市井故事，而非仅仅是网红打卡。
*   **沉浸式交互**：陪伴式导游，支持语音/文字实时交互，具备性格色彩的 AI 形象。

## 2. 市场痛点分析
*   **信息同质化**：传统平台推荐高度趋同，难以满足个性化、分众化的审美需求。
*   **导览缺失**：自由行用户在古迹或历史街区往往"看个热闹"，缺乏专业的即时讲解。
*   **过度工具化**：现有地图或旅游工具缺乏情感温度，交互割裂。

## 3. 核心功能模块

### 3.1 智能路线探索 (Smart Route Discovery)
*   **首页城市上下文**：顶部展示当前城市（首访根据浏览器定位解析；拒绝定位则默认北京）。用户可随时进入城市选择页手动切换。若用户**超过 7 天未访问**后再次打开首页，且定位解析到的城市与已选城市不一致，则弹出底部确认 sheet：主操作「切换至定位城市」、次要「保持当前城市」；点击遮罩或 Esc 视为保持，避免误触；同一「已选 ↔ 定位」组合在 7 天内若用户选择保持，则不再重复打扰（可再次手动切换城市）。
*   **主题化分类**：提供 Citywalk、老字号/中医药探秘、非遗体验、艺术漫游、自然探险等多元标签。
*   **动态热度系统**：基于实时大数据反馈路线受欢迎程度。
*   **微缩地图缩略**：不仅展示封面图，通过 SVG 动态生成路线路径预览，直观展示体力消耗与耗时。

### 3.2 AI 随行导游 (AI Companion)
*   **多性格人格化**：用户可选择不同语言风格的导游（如：地道京腔的老北京、专业的艺术馆馆长、亲切的旅行博主）。
*   **情境感知讲解**：基于 GPS 定位，当用户进入景点 30 米半径范围时，AI 自动推送相关背景知识（地理围栏触发）。
*   **多模态交互**：支持长按录音发送、文字输入，提供包含"知识卡片"、"景点卡片"、"美食推荐"在内的丰富交互内容。
*   **智能意图路由**：Agent 自动判断用户问题类型 —— 深度文化问题查询内部知识库（Local RAG），即时性问题（排队/票价/天气）调用外网搜索。

### 3.3 深度知识图谱 (Knowledge Repository)
*   **非遗活化**：深入挖掘如"兔儿爷"、"剪纸"等传统文化的历史脉络与制作工艺。
*   **传承人档案**：引介当地手工艺人、非遗传承人，为文化传播建立连接。
*   **RAG 驱动**：所有知识以结构化切片存储，AI 讲解时基于用户位置进行空间过滤 + 语义匹配 + 关键词增强的三重混合检索，确保"所见即所听"。

### 3.4 旅程回顾与收获 (Trip Summary)
*   **成就系统**：旅程结束后自动生成"收获卡片"，包含行走的里程、探索的知识点。
*   **打卡地图**：用户的全部打卡记录在地图上以发光勋章样式渲染，形成个人文化足迹。
*   **社交传播**：极简美学的海报生成功能，一键分享至社交平台。

## 4. 技术方案

> 完整的技术选型论证与架构设计详见 [WeGO_Technical_Solution.md](./WeGO_Technical_Solution.md)。以下为核心选型摘要。

### 4.1 前端与地图
*   **前端技术栈**：Vanilla JavaScript, CSS3, Semantic HTML5（Mobile-First）。
*   **地图引擎**：三图全接入（Mapbox / 高德 / 百度），通过统一 Adapter 抽象层屏蔽差异，用户可自主切换。
*   **坐标系统**：后端统一 WGS-84，前端使用 gcoord 库按引擎实时转换（GCJ-02 / BD-09）。
*   **跨平台策略**：Web 先行验证 -> Phase 2 使用 **Capacitor** 零改动打包为 iOS/Android App，支持后台定位与息屏播报。

### 4.2 AI 引擎与知识库
*   **Agent 编排**：采用 **LangGraph**（LangChain）实现多工具编排与状态管理。
*   **RAG 检索增强**：旅游知识按景点/主题切片，通过 Embedding 模型向量化后存入 **Supabase pgvector**，支持语义搜索 + 空间过滤 + 关键词增强的三重混合检索。
*   **LLM**：Claude Sonnet 4 / GPT-4o，支持复杂推理与人格化表达。
*   **AI 路线规划三步分离原则**：
    1.  AI 生成结构化 POI 节点序列（JSON）
    2.  LBS API 校验真实距离与耗时（防止 LLM 空间幻觉）
    3.  前端渲染真实轨迹线（Polyline）

### 4.3 后端与数据库
*   **BaaS 选型**：**Supabase**（PostgreSQL + pgvector + PostGIS + Auth + Realtime + Edge Functions），一库解决关系数据、向量检索与空间计算。
*   **实时推送**：通过 **SSE（Server-Sent Events）** 实现 Agent 讲解的流式推送。
*   **状态管理**：前端通过 `sessionStorage` 暂存对话进度，服务端通过 `agent_transcripts` 表持久化完整交互历史（JSONB 格式存储结构化卡片）。

## 4.4 路线数据生产闭环（生产环境）

为支撑持续上新路线，WeGO 在生产环境采用“非结构化内容到结构化入库”的标准闭环，确保内容质量、可追溯与可回滚：

1. **内容输入（原始稿件）**  
   运营或编辑提交路线 Markdown（可含图文、采访记录、口述材料）。原始稿件仅存档，不直接入库。
2. **结构化抽取（AI）**  
   Agent 将 Markdown 抽取为标准 `route + spots` JSON（包含标题、时长、标签、坐标、景点顺序等字段）。
3. **契约校验（Schema Gate）**  
   严格执行 `contracts/` 下 JSON Schema 校验；不符合契约的数据不可进入下一阶段。
4. **数据清洗（Quality Gate）**  
   执行字段标准化（单位、标签映射、数值类型、坐标范围、排序连续性、UUID 合法性）与业务规则校验。
5. **人工审核（Editorial Gate）**  
   审核展示文案、历史事实、封面图与路线体验逻辑，确认后进入发布队列。
6. **幂等入库（Publish Gate）**  
   通过后端服务事务化 upsert `routes` 与 `spots`，写入版本号与审计日志。
7. **发布回看（Operate Gate）**  
   发布后支持数据核验、异常回滚、二次修订。

该闭环将“内容生产能力”纳入产品核心能力，目标是让路线扩充速度与数据可靠性同时提升。

## 4.5 路线内容管理后台

### 4.5.1 产品定位

路线内容管理后台（Admin）是 WeGO 内容运营团队（内容审核人员）专用的内容管理工具，与面向终端用户的前台产品共用同一个 Supabase 实例，但通过独立的页面（`admin-routes.html`）和管理端 API SDK（`src/lib/admin-api.js`）实现权限物理隔离。

**与前台产品的关系**：
- 共用 Supabase 数据库实例（routes、spots 等核心表）
- 前台产品使用 Anon Key（公开读取）
- 管理后台使用 Service Role Key（完整读写权限，专属管理操作）
- 路由/景点数据变更后，前台产品实时可见（无缓存层）

### 4.5.2 功能范围

| 功能 | 说明 |
|---|---|
| **查看路线列表** | 按名称模糊搜索、category 过滤、分页（已移除 difficulty） |
| **查看路线详情** | 点击展开，查看该路线下所有游玩点卡片；展示可见性/彩蛋标记 |
| **新增路线** | 创建空白路线后进入编辑；编辑过程 **debounce 自动保存草稿** |
| **编辑路线** | title、description、tags、category、cover_image、路线是否可见；**时长/距离/形状缩略图**由系统按可见点与排序自动重算 |
| **发布** | 将当前草稿写入 **`route_versions` 快照**，递增 **published_version**，记录 **last_published_at** |
| **删除路线** | 确认后删除（级联删除关联游玩点与版本快照） |
| **编辑游玩点** | name、subtitle、short_desc、**rich_content**（图文，与 spot_type 模板配合）、detail（与 rich 同步）、tags、thumb、photos、坐标（可选 WGS-84 / GCJ-02 录入）、geofence、停留、排序、**spot_type**、**is_visible**、**is_easter_egg** |
| **删除游玩点** | 确认后删除单条 |

详细规则见 `docs/product-design/admin-route-editor.md`。

### 4.5.3 热度字段策略

`heat_level`（热度点数 0-5）和 `heat_count`（热度累计值）在管理后台中**仅展示，不可编辑**。这两个字段由用户在前台完成打卡后系统自动统计更新（参见 Sprint 6 打卡系统）。管理员如需调整，必须通过前台真实用户行为触发。

### 4.5.4 非结构化数据承载

游玩点图文以 **`rich_content`** 为主字段（各 `spot_type` 初期共用结构，前台按类型换模板）；`detail` 与导入链路兼容，可与 `rich_content` 同步。标签仍用 `tags[]`。后续如需元数据（来源 URL、编辑历史等），可扩展 `spots.metadata` JSONB。

## 4.6 Agent 自助路线上传

### 4.6.1 产品定位

路线内容是 WeGO 的核心资产，但其来源目前依赖人工 seed 脚本（见 4.4 节），无法满足持续扩充路线的需求。Agent 自助路线上传功能让运营人员或普通用户通过对话或上传页面自助提交路线内容，Agent 自动解析、结构化、校验、补全缺失内容，经人工二次确认后写入数据库。

**与 Sprint 9 数据清洗流水线的区别**：

| 维度 | Sprint 9（人工脚本流水线） | Sprint 11（Agent 自助上传） |
|:---|:---|:---|
| **触发方式** | 运营人员手动执行脚本 | 用户/运营通过对话或上传页随时触发 |
| **格式支持** | 仅 JSON | JSON / Markdown / TXT / URL 抓取 |
| **Gap 处理** | 人工介入修正 JSON 后重新执行 | Agent 自动处理 + 对话询问 |
| **补全能力** | 无 | 自动查询经纬度等客观数据 |
| **用户角色** | 仅运营 | 运营 + 普通用户 |
| **实时性** | 批量导入 | 单条实时处理 |

两者互补：Sprint 9 适合批量迁移历史数据，Sprint 11 适合日常新增单条路线。

### 4.6.2 核心流程

```
用户上传（JSON / Markdown / TXT / URL）
    ↓
Agent 解析 → 识别路线结构和景点列表
    ↓
逐 Spot 检查字段完整性 → 发现 Gap（缺失内容）
    ↓
┌─ 客观 Gap（经纬度 / 标签 / 封面图）
│   → Agent 调用高德 Geocoding API 自动查询
│   → 展示结果给用户确认后采纳
└─ 主观 Gap（副标题 / 详细介绍 / 短描述）
    → Agent 以对话方式询问用户
    → 用户补充后 Agent 验证并采纳
    ↓
所有必填项处理完毕 → 生成 route-ingestion 格式预览
    ↓
用户二次确认（预览报告）
    ↓
写入数据库（routes 表 + spots 表）
状态变更为 active → 上线
```

### 4.6.3 上传入口

支持三个入口，满足不同用户习惯：

**入口一：AI 对话入口**
触发词：「上传路线」「导入路线」「我想添加一条路线」
用户在 AI 聊天界面输入触发词并附加文件，Agent 接管后续全流程。

**入口二：独立上传页面**
路径：`/upload-route`（可通过底部导航或设置页进入）
提供文件上传（拖拽/点击）、URL 抓取、纯文本粘贴三种模式，适合非对话场景。

**入口三：URL 抓取**
用户粘贴马蜂窝/小红书/微信公众号等内容的 URL，Agent 自动抓取并解析页面正文，提取路线信息。

### 4.6.4 缺失内容（Gap）处理策略

| Gap 类型 | 字段 | Agent 处理方式 |
|:---|:---|:---|
| **客观 Gap** | `lat` / `lng`（经纬度） | 调用高德 Geocoding API 查询，展示结果给用户确认后采纳 |
| **客观 Gap** | `tags`（标签） | Agent 推断标签候选，展示给用户确认后采纳 |
| **客观 Gap** | `cover_image`（封面图） | Agent 搜索配图 URL，展示给用户确认后采纳 |
| **客观 Gap** | `estimated_stay_min`（建议停留） | 规则库 + LLM 推断，直接采纳（用户可后续修改） |
| **主观 Gap** | `subtitle`（副标题） | Agent 对话询问用户，用户补充后采纳 |
| **主观 Gap** | `detail`（详细介绍） | Agent 对话询问用户，用户补充后采纳 |
| **主观 Gap** | `short_desc`（短描述） | Agent 对话询问用户，用户补充后采纳 |

### 4.6.5 二次确认机制

Agent 完成解析后，以结构化预览报告形式呈现完整路线内容，用户必须点击「确认上传」方可写入数据库。

**预览报告内容**：
- 路线基本信息（标题 / 描述 / 分类 / 时长 / 距离 / 标签 / 可见性等）
- 游玩点列表（每个点的字段完整性和来源标注）
- 已自动查询补全的字段（标注"已自动查询"）
- 待用户补充的字段（标注"待补充"）

**用户操作选项**：
- **确认上传**：写入数据库，状态变更为 active
- **继续编辑**：返回 Gap 补全对话，继续补充缺失内容
- **取消上传**：放弃本次上传，保留草稿

### 4.6.6 状态机

```
draft（草稿） → parsing（解析中） → gap_filling（缺失补充）
                                              ↓
                               所有必填 Gap 均已处理完毕
                                              ↓
                                    pending_review（待确认）
                                              ↓
                         ┌─────────────────────┴─────────────────────┐
                         ↓                                           ↓
                confirmed（已确认）                            editing（用户编辑）
                         ↓                                           ↓
               写入数据库 → active（上线）←────────────────┘
```

### 4.6.7 上传文件格式示例

**JSON 标准格式**（符合 `route.schema.json` / `route-ingestion.schema.json`）：
```json

  "title": "京城胡同游",
  "description": "从杨竹梅斜街到炭儿胡同，感受老北京的烟火气",
  "duration_minutes": 180,
  "category": "文化",
  "tags": ["胡同", "文化"],
  "cover_image": "https://example.com/cover.jpg",
  "spots": [
    {
      "name": "杨竹梅斜街",
      "lat": 39.8973,
      "lng": 116.3976,
```
