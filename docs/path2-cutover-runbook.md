# WeGO 路径2灰度切流与回滚

## 验收清单

- 首页路线列表可读
- 路线详情 + 景点排序正确
- 打卡写入成功并可查询
- 路线上传：上传 -> gap-reply -> confirm 全链路通过
- 管理端：路线 CRUD + 发布 + 轮播配置
- Agent 查询知识接口可返回结果

## 100 在线压测

```bash
# 先准备一个普通用户 access token
BASE_URL=http://127.0.0.1:8787 ACCESS_TOKEN=<token> CONCURRENCY=100 DURATION_MS=60000 node scripts/load-test-100.js
```

通过标准（建议）：

- 失败率 < 5%
- 接口无持续 5xx
- PostgreSQL CPU/IO 未持续顶满

## 灰度切流步骤

1. 先在预发环境完成全量验收。
2. 线上保持旧入口，新增 `/api` 新后端。
3. 先让管理员使用新上传链路，观察 24 小时。
4. 前台 10% 用户切到新 `apiBaseUrl`。
5. 若核心指标稳定（成功率、时延、错误率）再全量。

## 回滚策略

- 回滚入口：将 `apiBaseUrl` 指回旧链路配置。
- 数据回滚：保留每日快照 + `pg_dump`，按时间点恢复。
- Agent 回滚：将 `BACKEND_API_URL` 指回旧服务或停用上传功能。

## 事故处理优先级

1. 上传确认失败（影响运营）  
2. 路由读取失败（影响用户主流程）  
3. 实时推送延迟（次要）  
4. 管理端非核心功能（可降级）
