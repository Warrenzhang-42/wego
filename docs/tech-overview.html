整体架构 — 三层结构（当前实际部署版）

WeGO 已迁移到完全自建的云服务器部署，由「用户手机端」「自建后端服务」「AI 智能核心」三层构成。

📌 为什么技术文档里提到 Supabase，但实际没在用？

WeGO 在早期规划阶段选用了 Supabase 作为一站式后端平台。但在实际部署上线后，团队改为在云服务器上自建所有服务，好处是成本更低、数据完全自主控制、不依赖第三方平台限制。

原来计划用 Supabase 提供的功能，现在都由自己的代码实现了：

地图引擎 — 三地图全接入

WeGO 同时支持高德、Mapbox、百度三款地图。架构上使用「适配器模式」统一调用，业务代码不感知差异。

🟠
高德地图（默认推荐）

国内 POI 数据最全，步行导航精度最高，合规备案。WeGO 默认使用。

GCJ-02 坐标系
🔵
Mapbox（设计感版）

视觉定制能力最强，支持 Studio 自定义样式。追求设计感的用户可选。

WGS-84 坐标系
🔴
百度地图（街景版）

街景数据丰富，国内用户基数大。需要街景能力时使用。

BD-09 坐标系

⚠️ 最难解决的问题：坐标系不统一

三家地图用的是不同的经纬度标准，同一个地点在三套坐标系里的数字不一样，直接混用会导致「标记偏移」（标记打在了河里、墙上）。WeGO 的解决方案：

  • 数据库统一存储国际标准 WGS-84（GPS 原始坐标）
  • 前端地图适配器（Adaptors）在渲染时根据当前底图需求实时转换坐标（如高德 GCJ-02，百度 BD-09）
  • 业务代码只需要处理 WGS-84 坐标，底层适配器自动抹平差异，产品侧完全透明

🔌 适配器模式：统一接口设计

不管底层用哪家地图，业务代码只调用统一的"WeGO 地图接口"，切换地图引擎时无需修改任何业务逻辑。

map.setCenter(lng, lat) // 移动地图中心
map.addMarker(lng, lat, opts) // 添加景点标记（自动坐标转换）
map.drawRoute(coords) // 绘制步行路线
map.addGeofence(lng, lat, r) // 添加地理围栏
map.switchProvider('amap') // 实时切换地图引擎

AI 导览 Agent — 大脑怎么工作

WeGO 使用 LangGraph 框架搭建多工具 AI Agent，能根据问题类型自动选择合适的工具作答。

🧠 什么是「Agent」？

普通 AI 像计算器——你问什么它就答什么。
WeGO 的 Agent 更像一位有工具箱的助理。它收到问题后会先判断——这个问题需要「查内部知识库」还是「搜互联网」还是「规划地图路线」——然后主动调用工具获取答案，最后综合生成回复。

Agent 的工具箱（7 个工具）

search_knowledge()
在本地知识库中语义检索景点故事、历史文化内容。结合用户所在位置，只返回附近 500 米内的知识。
📚 数据源：PostgreSQL 知识库向量表（pgvector）
web_search()
搜索最新的票价、营业时间、排队状况等即时信息，补充知识库里没有的实时内容。
🌐 数据源：Tavily Search API (无 Key 时自动降级 Mock)
plan_route()
根据 AI 选出的景点列表，调用地图 API 规划最优步行路线，并校验体力/时间约束。
🗺 数据源：高德地图 Directions API
geofence_narration()
当用户进入景点围栏时自动触发，拉取该景点的深度知识，生成主动讲解内容。
📍 触发类型：geofence（系统触发）
upload_route()
解析用户上传的路线文件（JSON / Markdown / 文本 / URL），自动补全缺失字段，检测信息缺口。
📤 输入：任意格式路线文档
confirm_route_upload()
用户二次确认路线数据无误后，执行数据库写入（routes + spots），完成闭环。
✅ 输出：写入 PostgreSQL 数据库（自建）
auto_query()
自动查询景点坐标（调用高德地理编码 API）、推断标签类型和建议停留时长，直接帮用户填信息。
🔍 辅助工具：非对话工具，内部调用

📖 RAG 是什么？用一句话解释

数据库 — 所有数据存在哪里

WeGO 在云服务器上自建 PostgreSQL，安装 pgvector、PostGIS、pg_trgm 三个插件，一个数据库同时承担关系存储、向量检索、空间计算三大功能。

🎯 为什么选择自建 PostgreSQL（而非 Supabase 托管）？

WeGO 最终选择在自己的云服务器上安装 PostgreSQL，核心原则依然是「一库搞定」：

  • PostgreSQL：成熟可靠的关系数据库，存路线、景点、用户等结构化数据。数据完全在自己服务器上，不依赖第三方。
  • pgvector 插件：在 PostgreSQL 里直接做知识库向量搜索，无需单独部署向量数据库。
  • PostGIS 插件：在 PostgreSQL 里直接做地理空间计算（比如「找出半径 500 米内的知识」）。
  • pg_trgm 插件：支持中文模糊搜索，路线标题搜索功能的基础。
  • 自建 API 服务：用 Express.js 写所有接口，完全自主控制，无平台锁定风险。

💡 与 Supabase 相比：自建方案成本更低（国内云服务器性价比高），国内访问延迟更小，且数据不出境，合规更安全。

地理围栏 — 走近景点自动讲解

WeGO 最核心的体验：用户不需要主动问，走进景点 30 米范围，AI 自动开始讲解。

🎯 什么是地理围栏？

想象每个景点周围有一个看不见的圆圈（默认半径 30 米）。当用户的 GPS 坐标进入这个圆圈，系统就触发事件。WeGO 用这个机制实现「走到哪里、讲到哪里」。

🔄 地理围栏触发全流程（6 步）

1

持续 GPS 监听

用户进入路线详情页后，前端通过 navigator.geolocation.watchPosition() 开启高精度监听，当设备位置变动时自动触发回调更新坐标。

2

计算与每个景点的距离

用 Haversine 公式（地球曲面距离公式）计算用户当前坐标与路线上每个景点坐标的直线距离。

3

检测是否进入围栏

若距离 ＜ 围栏半径（默认 30 米），系统会自动触发讲解。具备 5 分钟冷却机制：短暂离开后重新进入不会重复播报，确保游览体验流畅。

4

向 AI Agent 发送系统触发信号

前端构造携带 trigger_type: "geofence" 和 spot_id 的请求，发送给 Agent 服务。

5

Agent 生成专属讲解

Agent 调用 geofence_narration 工具 → search_knowledge(spot_id) 获取深度知识 → 结合用户选择的导游人格，生成口语化讲解内容。

6

流式推送到用户界面

讲解内容以 SSE 流式方式逐字推送至前端对话界面，同时展示知识卡片。整次讲解记录写入 agent_transcripts 表（trigger_type = 'geofence'）。

运营后台 — 内容团队怎么管理内容

admin-routes.html 是内容运营人员专用的管理工具，直接用浏览器打开，无需服务器部署。

前台用户使用「只读密钥」，只能查看数据。运营后台使用「管理员密钥」，可以增删改路线和景点。两套密钥完全隔离，互不影响。

📸

运营可在后台调整首页大图轮播内容，支持全站通用配置或按城市覆盖配置。修改即时生效，无需等待开发。

📦

每次「发布」路线都会保存一份完整快照（route_versions 表）。可以追溯历史版本，误操作可以回滚，数据安全。

🤖

AI 辅助路线入库

运营只需上传 Markdown 文稿，AI 自动解析结构、补全坐标、识别信息缺口，运营确认后一键写库。大幅降低数据录入成本。

📝 路线上传流程（AI 辅助入库）

①

上传原始文稿

运营上传 JSON / Markdown / 纯文本 / URL，或直接在 AI 对话框里描述路线。

②

AI 自动解析 & 补全

Agent 自动提取路线名称、景点列表、停留时长，调用高德地理编码 API 查询景点坐标，自动推断分类标签。

③

展示缺口 & 用户补充

若有无法自动填写的主观信息（如路线故事简介），AI 会列出「信息缺口」，询问用户补充。

④

用户二次确认，写入数据库

运营确认数据无误后，AI 执行原子事务：routes 表 + spots 表同时写入，同时保存审计日志，失败自动回滚。

产品经理常见疑问

当你向技术团队问出这些问题，你就是一个真正懂技术的 PM。

AI 会「乱说」吗？用了什么机制防止幻觉？
WeGO 用 RAG（检索增强生成）来减少幻觉：AI 在回答前会先去本地知识库找相关内容，基于找到的真实资料生成回答，而不是凭空编造。 对于知识库里没有的内容（如即时票价），会转而调用 web_search 获取实时信息，而不是瞎猜。 当然，RAG 不能 100% 消除幻觉，还需要持续维护知识库的内容质量。
用户没有网络时还能用吗？
目前 WeGO 大部分功能依赖网络：地图瓦片加载、AI 对话、知识库检索都需要联网。 离线支持是 Phase 2+ 的规划，届时可以预缓存路线数据和知识内容到本地，实现部分离线体验。 目前建议在景区入口提前打开路线详情页，让地图预加载完成。
数据安全吗？用户隐私如何保护？
GPS 坐标仅在会话期间使用，仅用于地理围栏判断，不持久化存储用户实时轨迹。 系统采用自建 JWT 认证（bcryptjs 加密密码 + jsonwebtoken 签发令牌），数据库通过 app_users 表的 role 字段区分普通用户和管理员权限，接口层通过 requireAuth / requireAdmin 中间件拦截。 对话记录（agent_transcripts）关联了 user_id，将来会实现用户可以删除自己的对话记录。
AI 响应速度慢怎么办？用户等待体验如何保证？
主要通过「流式输出（SSE）」优化体验：AI 不是等全部生成完再返回，而是一个字一个字地推送给前端，用户会看到文字逐渐出现，感知上比一次性返回快很多。 技术上实现了「自动降级机制」：若主模型（定义在 env 中）连接超时或失败，会自动切换到备用模型重试（如从 GPT-4 降级到 DeepSeek 或 MiniMax），确保回复不中断。 长期优化方向：知识库检索结果预热缓存、热门问题答案缓存等。
运营想新增一条路线，需要开发参与吗？
设计目标是：普通运营人员可以独立完成路线入库，无需开发介入。 路线可以以 Markdown / JSON / 纯文本上传，AI 自动解析并补全坐标、标签等信息，运营确认后一键写库。 但目前管理后台 UI 还在完善阶段，复杂路线仍建议技术 review 后再发布。
地理围栏精度有多高？会不会到了景点 AI 没触发？
GPS 定位精度在开阔环境下约 ±5 米，但城市高层建筑环境可能误差到 ±20-30 米。 WeGO 默认围栏半径 30 米，可以在后台按景点调整。对于室内景点（如展厅、门店），可根据实际情况增大到 50-80 米。 城市 GPS 也会受「卫星信号反射」影响，在高楼峡谷中偶发偏移是正常现象，已在技术文档中标注为已知限制。
地图切换（高德 / Mapbox / 百度）对用户体验有什么影响？
三套引擎使用统一的「适配器接口」，切换时路线数据和景点标记都会保留，只是底图样式和渲染风格不同。 切换地图引擎需要重新加载对应 SDK，可能有 1-3 秒的空白过渡。 目前默认高德，国内用户体验最稳定。Mapbox 在国内部分地区加载较慢（CDN 缘故），这是已知限制。
